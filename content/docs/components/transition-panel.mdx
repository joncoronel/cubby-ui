### Vertical Axis

Slide views along the Y axis instead of X. Useful for stepped flows where movement should feel like turning a page rather than navigating left/right. The container's height also animates as the active view's content size changes.

## API Reference

`TransitionPanel` is a pure-CSS compound component — there is no underlying Base UI primitive. Views register themselves through context, so `TransitionPanelView` does not need to be a direct child: wrappers, fragments, conditional rendering, and Suspense boundaries are all supported.

The entrance animation uses `@starting-style` and `transition-behavior: allow-discrete`. Older browsers degrade to an instant swap with no slide — no broken state, just no motion. The component also respects `prefers-reduced-motion`.

### Props

#### TransitionPanel

Outer container. Renders a `<div>` and accepts any native div prop (`id`, `role`, `aria-label`, `ref`, `data-*`, `style`, etc.). Composes a consumer `ref` with the internal ref used for height measurement.

<ApiPropsList>

<ApiProp
  name="activeKey"
  fullType="string"
  required
>
  The `viewKey` of the currently visible view. Must match the `viewKey` of one of the `TransitionPanelView` descendants. Changing this value triggers the slide and height animation.
</ApiProp>

<ApiProp
  name="axis"
  fullType='"x" | "y"'
  simpleType="string"
  defaultValue='"x"'
>
  Slide axis for the view transition. `"x"` slides horizontally; `"y"` slides vertically.
</ApiProp>

</ApiPropsList>

##### CSS custom properties

Set on the root and inherited by view wrappers. Override via the `style` prop or a CSS rule.

<ApiPropsList>

<ApiProp
  name="--tp-duration"
  fullType="time"
  defaultValue="240ms"
>
  Duration of both the height transition and the slide / opacity transition on views.
</ApiProp>

<ApiProp
  name="--tp-clip-margin"
  fullType="length"
  defaultValue="0px"
>
  Bleed area for the panel's overflow clip. Defaults to `0` because the recommended composition (frame on the panel, padding inside each view) keeps focus rings and shadows well inside the clip box. Bump it (e.g. `8px`) only when the panel is nested inside a padded container where view content sits flush with the clip edge.
</ApiProp>

</ApiPropsList>

#### TransitionPanelView

A single view registered with the parent panel. Renders a `<div>` and accepts any native div prop (`className`, `style`, `id`, `aria-label`, `aria-labelledby`, `ref`, `data-*`, etc.) in addition to those documented below — spread props land on the view wrapper. The consumer `ref` is composed with the internal ref used for registration with the parent panel. Visibility, slide offset, and `inert` state are driven by the parent's `activeKey`. Inactive views stay mounted (their internal state survives swaps) but are marked `inert` and `aria-hidden` so focus and assistive tech skip them.

<ApiPropsList>

<ApiProp
  name="viewKey"
  fullType="string"
  required
>
  Identifier matched against the parent's `activeKey` to determine when this view is visible.
</ApiProp>

<ApiProp
  name="initialFocus"
  fullType="boolean | React.RefObject<HTMLElement | null>"
  defaultValue="true"
>
  What to focus when this view becomes active after a swap. `true` focuses the first tabbable element inside the view; `false` opts out; a ref focuses that specific element. Skipped on the initial render — the parent surface (popover, dialog, sheet, page) owns first-mount focus. Focus is applied with `{ preventScroll: true }` so the entrance animation can finish without scroll-jank.
</ApiProp>

</ApiPropsList>

### Notes

- **Composability.** Views can be wrapped in HOCs, fragments, error boundaries, Suspense boundaries, or rendered via `.map()` and conditional logic — registration routes through context, not React children.
- **Direction inference.** Direction is computed by comparing the new and previous `activeKey` against the registered view order, so back-navigation slides the opposite way automatically. Re-deriving from DOM position on each registration means a view that unmounts and remounts (e.g. `{isPro && <TransitionPanelView ...>}`) keeps its source-order slot.
- **Focus management.** Focus moves in a `useLayoutEffect`, before the browser paints, to minimize the gap between the outgoing view becoming `inert` and the incoming view receiving focus.
- **Height animation.** Animates the real `height` property (via `ResizeObserver`) rather than `transform: scale`, so descendant layout doesn't warp. Acceptable for typical panel use cases; if you need compositor-only height animation for a high-frequency surface, fork to a Motion / FLIP implementation.
